## 🔴 КРИТИЧЕСКИЕ (Critical)

### C1. RLS-политики Supabase не проверяемы и, вероятно, неполны — **всё ещё открыто**
- **Где:** `supabase/migrations/` содержит только `add_crypto_keys_to_shops.sql`. Политики RLS для таблиц `profiles`, `items`, `receipts`, `business_receipts`, `shops`, `waitlist` **отсутствуют в репозитории**.
- **Риск:** Весь клиентский код полагается на RLS как единственный контроль доступа (anon-ключ публичен). При некорректных RLS: утечка `private_key` всех магазинов (см. C2); чтение чужих `receipts`, `items`, `profiles`.
- **Действие:** Опубликовать/проверить RLS для каждой таблицы в Supabase Dashboard. Это требует доступа к самому проекту Supabase — не исправляется правкой кода в репозитории.

### C2. Приватные ключи Ed25519 хранятся в открытом виде — **всё ещё открыто**
- **Где:** `js/business-panel.js` (insert `private_key` в таблицу `shops`; кэш в `sessionStorage`).
- **Риск:** Приватный ключ магазина лежит в БД в plaintext. Любой XSS или утечка БД (см. C1) → полная компрометация подписей магазина.
- **Действие:** Не хранить `private_key` в таблице, доступной по RLS кому-либо кроме владельца. Архитектурное решение, требует отдельной задачи.

### C3. Несовпадение формата данных при проверке подписи — **✅ ИСПРАВЛЕНО (08.07.2026)**
- **Было:** подпись строилась из сырых данных формы (`fd.get('purchase_date')` вида `"2026-07-07T12:00"`, необработанные float), проверка — из данных БД (другой формат даты, другая точность чисел) → `verify()` возвращал `false` для любого чека, включая подлинные.
- **Исправлено:** добавлена `buildSignaturePayload()` в `js/crypto-signature.js` — единая функция канонизации (дата → `YYYY-MM-DD`, суммы → `toFixed(2)`), используется и при подписи (`business-panel.js`), и при проверке (`crypto-verification.js`). Проверено тестом: одинаковый чек с «сырыми» и «из БД» значениями теперь даёт идентичную строку для подписи.

### H2. Слабая политика паролей при регистрации
- **Где:** `js/register.js` (только `password.length < 6`); при этом `js/security.js` `validatePassword` требует 8+ символов, заглавные/строчные/цифру/спецсимвол, но нигде не вызывается при регистрации.

- **Действие:** Добавить Turnstile.

### H4. Rate-limit только на клиенте
- **Где:** `js/security.js` (`Map` в памяти браузера) — легко обходится.
- **Действие:** Серверный rate-limit (Supabase/Edge Function).

### H5. Загрузка файлов без проверки типа
- **Где:** `js/receipts.js` — проверка только размера, `accept` — клиентский, обходится.
- **Действие:** Проверка MIME на стороне storage, запрет `text/html`/`image/svg+xml`.

### H6. Неполное удаление аккаунта
- **Где:** `js/dashboard-settings.js` — не удаляет `auth.users`, не чистит `business_receipts` с `customer_email` пользователя, `storage.list` не пагинирован (> 100 файлов не удаляются).
- **Действие:** Edge Function с `auth.admin.deleteUser` + каскад.

## 🆕 НОВОЕ (найдено 08.07.2026)

### N1. `[data-animate]`-элементы, добавленные в DOM динамически, никогда не появлялись — **✅ ИСПРАВЛЕНО**
- **Симптом:** сообщение «нет чеков — создать чек» и другие пустые состояния мелькали на долю секунды и пропадали навсегда (это и есть баг, из-за которого написан этот пункт аудита).
- **Причина:** `js/animations.js` сканировал `[data-animate]` один раз при `DOMContentLoaded` и наблюдал только уже существующие в этот момент элементы. Пустые состояния в `js/dashboard-items.js` и `js/receipts.js` рендерятся позже через `innerHTML` с атрибутом `data-animate="zoom"` — CSS сразу ставит им `opacity: 0`, но `IntersectionObserver` никогда их не видел, `.is-visible` не добавлялся никогда.
- **Исправлено:** `animations.js` переписан — теперь держит общий `IntersectionObserver`, и `MutationObserver` на `document.body` подхватывает любые новые `[data-animate]`-элементы, появившиеся когда угодно после загрузки страницы, и подключает их к тому же наблюдателю. Работает для всех текущих и будущих мест использования `data-animate` на динамическом контенте, не только для конкретных пустых состояний.

### N2. Иконки исчезали с двух кнопок на главной — **✅ ИСПРАВЛЕНО**
- **Где:** `index.html`, кнопки «Начать бесплатно» и «Открыть магазин».
- **Причина:** тот же класс бага, что и N1/старые баги со статусом чека — `data-i18n` висел на элементе, который также содержал `<i>`-иконку; `el.innerHTML = translations[...][key]` при каждой загрузке/смене языка стирал иконку.
- **Исправлено:** иконка вынесена за пределы переводимого элемента (обёрнут в отдельный `<span data-i18n="...">`).

### N3. Битая разметка `login.html` — **✅ ИСПРАВЛЕНО**
- Лишний `</div>` без пары — удалён.

### N4. Кнопка «Запросить новую ссылку» ничего не запрашивала — **✅ ИСПРАВЛЕНО**
- **Где:** `reset-password.html` + `js/auth.js`.
- Раньше просто разлогинивала и молча кидала на `login.html`. Теперь редиректит на `login.html?forgot=1`, который сам открывает модалку восстановления пароля.

### N5. `setSession` без обработки ошибок на `reset-password.html` — **✅ ИСПРАВЛЕНО**
- При истёкшем/невалидном токене страница раньше молча ломалась (форма видна, но не работает). Обёрнуто в try/catch, показывает экран ошибки.

### N6. Восстановление пароля не проверяло, зарегистрирован ли email — **✅ ИСПРАВЛЕНО (по запросу)**
- **Где:** `js/auth.js`, форма восстановления пароля.
- Добавлена проверка через существующую RPC `check_profile_exists` (та же, что используется в бизнес-панели) — если email не зарегистрирован, показывается предупреждение вместо тихой отправки письма в никуда.
- **⚠️ Важно понимать компромисс:** Supabase `resetPasswordForEmail` по умолчанию **намеренно** не сообщает, существует ли email — это защита от user enumeration (перебора email-адресов для фишинга/подбора). Этот фикс осознанно жертвует этой защитой ради лучшего UX, по прямому запросу. Смягчает риск существующий rate-limit на попытки (`checkLoginRateLimit`), но полноценной защиты от массового перебора email на клиенте нет (см. H4).

### N7. Мобильное меню не закрывалось по Escape — **✅ ИСПРАВЛЕНО**
- **Где:** `js/dashboard-nav.js` (dashboard.html), инлайн-скрипт `receipts.html`. Добавлено закрытие по Escape (уже было по клику на оверлей).
