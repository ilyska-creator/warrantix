## EPIC B — Надёжность и безопасность (из вашего же аудита)

### B-1 🔴 Реальная Ed25519-подпись через WebCrypto
- **Проблема:** в презентации помечено как «🚧 в разработке» — а это главный УТП продукта.
- **Что сделать:** генерация keypair на стороне продавца (Supabase Edge Function или клиент + защищённое хранение приватного ключа), подпись payload чека, проверка подписи в `verify.js` через публичный ключ.
- **Acceptance criteria:** чек, подписанный поддельным/чужим ключом, не проходит верификацию на `verify.html`.
- **Сложность:** L

### B-2 🔴 Аудит и допиливание RLS-политик Supabase
- Перед реальными пользователями обязательна ревизия: кто может читать/писать чужие чеки, магазины, профили.
- **Сложность:** M

### B-3 🔴 Убрать дублирование Supabase-клиента (ARCH-1)
- Клиент создаётся в 5 разных файлах (`auth.js`, `dashboard-auth.js`, `register.js`, `script.js`, `verify.js`) — ротация ключа требует правки везде, риск рассинхрона.
- **Что сделать:** вынести в один `js/supabase-client.js`, импортировать везде.
- **Сложность:** S

### B-4 🟠 Закрыть XSS-вектор в `setSafeHTML` (MED-7)
- `allowHTML=true` при пользовательском вводе — точка для инъекции.
- **Сложность:** S

### B-5 🟠 Закрыть XSS-вектор в переводах через innerHTML (MED-9)
- `script.js:132`, `login-lang.js:127` — сами тексты сейчас статичные, но паттерн опасен на будущее (особенно если переводы когда-то станут редактируемыми).
- **Сложность:** S

### B-6 🟠 SRI/integrity для всех CDN-скриптов (MED-13)
- FontAwesome, jsPDF, qrcode-generator и т.д. подключены без integrity-хэша — риск supply-chain атаки.
- **Сложность:** S

### B-7 🟡 QR-формат: экранировать `|` и `:` в значениях (MED-8)
- Ломает парсинг чека, если в данных встречаются эти символы (например, в названии товара).
- **Сложность:** S

### B-8 🟡 Донабить недостающие i18n-ключи (I18N-1, I18N-2, I18N-3)
- `verify_btn` отсутствует в `business-lang.js`; `msg_success` (EN) отсутствует в `script.js`; `security.js` шлёт сообщения только на русском.
- **Сложность:** S

## EPIC C — Доставка напоминаний (ядро ценности продукта)

### C-1 🔴 Email-доставка напоминаний о гарантии
- **Проблема:** `dashboard-notifications.js` сейчас только рисует список в UI — реальной отправки нет. Это критично: «напомним за 30/7/1 день» — главный крючок retention.
- **Что сделать:** Supabase Edge Function на cron (раз в день) → выборка item'ов, у которых до окончания гарантии 30/7/1 день → отправка через email-провайдер (Resend или Postmark, оба ок с EU).
- **Acceptance criteria:** тестовый item с истекающей через 7 дней гарантией реально получает письмо один раз (не дублируется при повторных запусках крона).
- **Сложность:** M

### C-2 🟠 Шаблон письма-напоминания
- Простой HTML-email с брендингом Valuon, ссылкой на конкретный item в дашборде.
- **Сложность:** S

### C-3 🟡 Web Push уведомления (опционально)
- Более высокий retention-эффект, но требует Service Worker — можно отложить до после первого пилота.
- **Сложность:** L

### C-4 🟡 Экспорт напоминания в календарь (.ics)
- Кнопка «Добавить в календарь» на карточке товара — снижает риск, что пользователь пропустит email.
- **Сложность:** S

## EPIC D — Локализация под Эстонию

### D-1 🔴 Добавить эстонский язык (`et`) в `translations`
- **Проблема:** сейчас только RU/EN, для рынка Эстонии это нетипично — государственный язык эстонский.
- **Что сделать:** добавить `et` в объект переводов в `script.js` (и во все `*-lang.js` файлы для остальных страниц), сделать его языком по умолчанию для `.ee`-трафика или по `navigator.language`.
- **Сложность:** M (перевод текста — S, если отдать переводчику/носителю; интеграция в код — S)

### D-2 🟠 Автоопределение языка по браузеру/гео при первом визите
- Сейчас по умолчанию всегда `ru`. Для Эстонии логичнее `et`/`en` по умолчанию, `ru` — по выбору.
- **Сложность:** S

### D-3 🟡 lang="ru" → правильный `lang` атрибут на каждой странице (LOW-1)
- Сейчас `lang="en"` захардкожен на страницах с русским контентом — плохо для SEO и accessibility, тем более с добавлением `et`.
- **Сложность:** S

## EPIC G — Аналитика и метрика Retention D30

### G-1 🔴 Трекинг событий для расчёта Retention D30
- **Проблема:** вы сами обозначили Retention D30 ≥ 25% как gate-метрику пилота, но сейчас нигде не трекается.
- **Что сделать:** минимально — таблица событий в Supabase (`user_id`, `event`, `created_at`) на ключевые действия: регистрация, добавление товара, открытие дашборда. Дашборд/SQL-запрос для расчёта retention.
- **Сложность:** M

### G-2 🟡 Продуктовая аналитика (PostHog/Plausible)
- Более удобная альтернатива самописному трекингу, если бюджет позволяет self-host или EU-cloud тариф (важно для GDPR).
- **Сложность:** S (интеграция) — но требует cookie-consent (A-3) для non-essential трекеров.
