## 1. Вёрстка — общие проверки (на КАЖДОЙ странице)

### Шапка / навигация
- [ ] Логотип не съезжает, не размывается при масштабе
- [ ] Меню на десктопе — все пункты в одну строку, не переносятся криво
- [ ] Бургер-меню на мобильном открывается/закрывается, не блокирует скролл фона намертво (проверить, что можно закрыть свайпом/тапом вне меню)
- [ ] `sidebar-overlay` в дашборде реально появляется под сайдбаром на мобильном (это баг из аудита — проверить визуально на реальном телефоне, не только в devtools)
- [ ] Активный пункт меню подсвечен корректно на каждой странице
- [ ] Тоггл темы (☀️/🌙) переключает мгновенно, без "мигания" белым/чёрным (FOUC)
- [ ] Тоггл языка меняет текст без сдвига layout (проверить, что EN-текст короче/длиннее RU не ломает кнопки/карточки)
- [ ] Хедер не перекрывает контент при скролле (sticky/fixed позиционирование)

### Общая типографика и отступы
- [ ] Ни один текст не обрезан/не наезжает на соседний элемент ни в одной локали
- [ ] Длинные email/названия магазинов/имена не переполняют карточки (проверить `overflow: hidden`/`text-overflow: ellipsis` или перенос)
- [ ] Все интерактивные элементы (кнопки, ссылки, чекбоксы) — не менее 44×44px на мобильном (тап-таргет)
- [ ] Между блоками нет "прыгающих" отступов при смене темы
- [ ] Placeholder-тексты в полях видны и не сливаются с фоном ни в одной теме
- [ ] Иконки (Font Awesome) прогружаются и не показывают "квадратик"/пустое место при медленном интернете

### Формы (все страницы с формами)
- [ ] Фокус на поле — виден чёткий outline (для клавиатурной навигации)
- [ ] Состояние ошибки поля — видно визуально (цвет рамки + текст ошибки), не только цветом (добавить иконку для дальтоников)
- [ ] Состояние успеха — видно явно
- [ ] Кнопка submit — показывает спиннер/disabled-состояние во время запроса, чтобы нельзя было дважды кликнуть (двойной сабмит)
- [ ] Автозаполнение браузера (autofill) не ломает стили полей (жёлтый фон Chrome autofill)
- [ ] На мобильном — правильный `inputmode`/`type` (email-клавиатура для email, числовая для сумм и т.д.)
- [ ] Zoom при фокусе на input на iOS не срабатывает нежелательно (шрифт полей ≥16px)

### Toast / модалки / оверлеи
- [ ] Toast-уведомления не перекрывают критичные кнопки/поля
- [ ] Toast исчезает сам по таймеру и по крестику
- [ ] Несколько toast подряд — не накладываются друг на друга, стекуются аккуратно
- [ ] Модальные окна: скролл фона блокируется, пока модалка открыта
- [ ] Модалка закрывается по клику вне, по Esc, по крестику — все 3 способа работают
- [ ] Модалка на мобильном не выходит за границы экрана, скроллится сама, если контента много
- [ ] `alert()` в генераторе PDF-чека (найдено в аудите) — заменить на toast, проверить что новый toast не блокирует поток как `alert()`

## 3. Функционал — полное покрытие по флоу

### Регистрация / вход / восстановление пароля
- [ ] Регистрация с валидным email — успешна, приходит подтверждение (если предусмотрено)
- [ ] Регистрация с уже существующим email — понятная ошибка, а не generic "что-то пошло не так"
- [ ] Регистрация с некорректным email (без @, без домена) — блокируется на клиенте до отправки
- [ ] Слабый пароль — блокируется с понятным объяснением требований (сейчас в коде только `length < 6` — проверить, действительно ли этого достаточно как UX-сообщения)
- [ ] Вход с верными данными — успешен, редиректит в дашборд
- [ ] Вход с неверным паролем — понятная ошибка, без утечки "email существует/не существует" (защита от enumeration)
- [ ] Rate-limit при множественных неудачных попытках входа — срабатывает, сообщение о времени ожидания корректно на ОБОИХ языках (найден баг: текст "мин" захардкожен на русском)
- [ ] Восстановление пароля — письмо приходит, ссылка рабочая, ссылка одноразовая/с истечением
- [ ] Смена пароля через reset-ссылку — работает, старый пароль после этого не подходит
- [ ] Logout — реально завершает сессию (проверить, что после logout нельзя вернуться назад кнопкой браузера и увидеть приватные данные)
- [ ] Истечение сессии (токен протух) — корректный редирект на login, а не белый экран/ошибка в консоли
- [ ] Captcha (Turnstile) — блокирует сабмит без прохождения, не даёт забайпасить через отключение JS

### Личный кабинет — чеки (покупатель)
- [ ] Загрузка чека (фото JPG/PNG) — сохраняется, отображается в списке
- [ ] Загрузка чека (PDF) — сохраняется, превью корректно
- [ ] Загрузка файла НЕ поддерживаемого формата (например .docx) — блокируется с понятным сообщением
- [ ] Загрузка файла больше лимита (>10MB, по коду) — блокируется с сообщением о лимите, а не тихо падает
- [ ] Ручное добавление чека (без файла, только данные) — сохраняется корректно
- [ ] Редактирование чека — сохранённые изменения применяются, отменённые (Cancel) — не применяются
- [ ] Удаление чека — запрашивает подтверждение, реально удаляется (и файл из Storage тоже, не только запись из БД)
- [ ] Двойной клик на "Сохранить" при редактировании — не создаёт дубликат (race condition, был в аудите)
- [ ] Гарантия: срок считается от даты покупки + месяцы корректно (проверить граничные случаи: 0 месяцев, отсутствие гарантии, конец месяца — напр. 31 января + 1 месяц)
- [ ] Статус гарантии (активна/истекает скоро/истекла) — совпадает между карточкой в "Мои вещи" и уведомлением (был баг рассинхрона в аудите)
- [ ] Поиск/фильтр по чекам — находит по названию магазина, товару, дате
- [ ] Экспорт/скачивание отдельного чека в PDF — открывается, все данные читаемы

### Уведомления
- [ ] Уведомление о скором истечении гарантии — приходит за корректное количество дней до истечения
- [ ] Отметка "прочитано" — работает, счётчик уменьшается
- [ ] Клик по уведомлению — ведёт на нужный чек/вещь
- [ ] Уведомления не дублируются при повторном рендере/обновлении страницы

### Бизнес-панель (магазин)
- [ ] Регистрация магазина (название, реквизиты, адрес) — сохраняется полностью
- [ ] Генерация ключевой пары Ed25519 — происходит один раз, при повторном заходе не перегенерируется случайно
- [ ] Выдача чека с ОДНОЙ позицией — подпись создаётся, чек появляется у покупателя (по email/номеру)
- [ ] Выдача чека с НЕСКОЛЬКИМИ позициями (multi-item) — сумма считается верно, каждая позиция сохраняется отдельно, гарантия по каждой позиции своя (проверить именно это — фича добавлена недавно)
- [ ] Отрицательная/нулевая цена товара — блокируется валидацией
- [ ] Отмена/аннулирование выданного чека — статус меняется, покупатель видит актуальный статус, QR такого чека при сканировании явно показывает "аннулирован" (не "подлинный")
- [ ] Выбор способа оплаты в чеке — сохраняется, отображается покупателю на понятном языке независимо от текущей локали интерфейса
- [ ] Экспорт списка выданных чеков (если есть) — данные полные, кодировка корректна при открытии в Excel (важно для кириллицы в CSV)

### Верификация чека (публичная, без логина)
- [ ] Сканирование валидного QR — показывает "Подлинный", все данные чека, включая товары
- [ ] Сканирование QR несуществующего/поддельного чека — показывает явную ошибку/предупреждение
- [ ] Сканирование QR аннулированного чека — явно показывает статус "аннулирован", НЕ "подлинный"
- [ ] Проверка чека, где подписаны не все поля (известный архитектурный пробел: `gross_total`, `email`, `payment_method`, `status` не входят в подпись) — убедиться, что интерфейс НЕ вводит пользователя в заблуждение, будто вся видимая информация криптографически подтверждена
- [ ] Повторное сканирование того же чека — не создаёт побочных эффектов (не плодит записи, не засчитывает повторно что-либо)

### Настройки профиля
- [ ] Смена email — требует подтверждения на новый адрес
- [ ] Смена пароля из настроек (не через reset) — требует ввода текущего пароля
- [ ] Удаление аккаунта — предупреждает о необратимости, реально удаляет данные (или анонимизирует, если так задумано), проверить каскадное удаление чеков/гарантий
- [ ] Смена языка интерфейса — сохраняется между сессиями (не сбрасывается при перезаходе)
- [ ] Смена темы — сохраняется между сессиями и между устройствами (если синхронизируется через профиль, а не только localStorage)

### Общие сквозные сценарии
- [ ] Полный E2E: регистрация → добавление магазина → выдача чека → чек появился у покупателя → покупатель отсканировал QR → верификация прошла успешно
- [ ] Работа без интернета / при потере соединения на середине действия (например, отправка формы) — понятная ошибка, а не зависание
- [ ] Работа на медленном 3G — все спиннеры/скелетоны показываются, ничего не выглядит "сломанным" из-за задержки
- [ ] Открытие ссылки на конкретный чек в новой вкладке / шеринг ссылки — работает без авторизации (если чек публично проверяем) или корректно требует логин (если приватный)
- [ ] Одновременное открытие приложения в 2 вкладках — изменения в одной подтягиваются или хотя бы не конфликтуют в другой

## 4. Переводы (RU ↔ EN) — построчная проверка

Метод: переключить язык на КАЖДОЙ странице и визуально пройти сверху вниз, ища:
- нетронутый текст (остался на другом языке или ключ `data-i18n="..."` вместо перевода);
- обрезанный/наехавший текст из-за разной длины RU/EN;
- некорректный грамматический род/число (частая проблема автоперевода: "у вас 1 чеков");
- неправильный формат даты (RU: `ДД.ММ.ГГГГ`, EN обычно `MM/DD/YYYY` или `DD Mon YYYY` — решить единый формат и проверить, что используется он, а не смешение);
- неправильный формат чисел (разделитель тысяч, десятичный разделитель — RU использует запятую, EN точку).

Файлы для сверки (проверить оба языка на каждый): `business-lang.js`, `dashboard-lang.js`, `login-lang.js`, `privacy-lang.js`, `roadmap-lang.js`, `terms-lang.js`, `verify-lang.js`, плюс переводы внутри `script.js` (для `index.html`).

- [ ] `index.html` — hero, badge, все 3+ карточки фич, футер, meta-теги (title/description на каждом языке для SEO)
- [ ] `login.html` / `register.html` — все лейблы полей, placeholder'ы, тексты ошибок валидации, текст капчи
- [ ] `reset-password.html` — все статусы (письмо отправлено / ссылка недействительна / пароль изменён)
- [ ] `dashboard.html` — меню, заголовки секций, статистика, пустые состояния, все toast-сообщения (успех/ошибка для каждого действия)
- [ ] `receipts.html` — карточки чеков, статусы гарантии (активна/истекает/истекла), модалка добавления/редактирования, все сообщения валидации формы
- [ ] `business.html` — форма магазина, форма выдачи чека, названия способов оплаты, статусы чеков (выдан/аннулирован)
- [ ] `verify.html` — статусы верификации (подлинный/поддельный/не найден/аннулирован), подсказки по сканированию, ошибки доступа к камере
- [ ] `privacy.html` / `terms.html` — юридический текст полностью переведён (не только заголовки, частая ошибка — machine-translate только шапки, тело на одном языке)
- [ ] `roadmap.html` — все пункты дорожной карты, статусы этапов
- [ ] Email-уведомления (если генерируются через Supabase/сторонний сервис) — тема письма и тело на языке пользователя, а не всегда на одном
- [ ] PDF-чек (генерируется `receipt-generator.js`) — все подписи полей на языке, в котором пользователь выдавал чек (и остаются ли они читаемы, если покупатель потом переключит интерфейс на другой язык — это разные вещи, проверить обе)
- [ ] Сообщение rate-limit при входе (`security.js`, `resetIn`) — отдельно проверить, сейчас захардкожено "N мин" вне зависимости от языка
- [ ] Валюта и её текстовое обозначение — единообразны в обеих локалях (и это тот же баг `$` вместо `€`, теперь ещё и в контексте переводов — стоит проверить, что при английской локали тоже не появляется `USD`)
- [ ] Название бренда/продукта — не переводится и не склоняется по падежам в RU-версии (частая ошибка автоперевода)
- [ ] Проверить `<html lang="...">` атрибут — меняется вместе с переключением языка (важно для screen reader и SEO)
- [ ] Проверить `og:title`/`og:description` (превью при шаринге в соцсети/мессенджеры) — на каком языке показываются, соответствует ли содержимому

## 7. Доступность (a11y) — быстрый проход

- [ ] Вся навигация проходима только клавиатурой (Tab/Shift+Tab/Enter), без мыши
- [ ] `<a href="#">`, используемые как кнопки (тоггл темы/языка, logout) — заменить на `<button>` или добавить `role="button"` + обработку Enter/Space
- [ ] Alt-тексты на всех значимых изображениях (не на декоративных — там `alt=""`)
- [ ] Контраст текста к фону — соответствует WCAG AA (особенно `--text-muted` на `--card-bg` в тёмной теме)
- [ ] Статусы (успех/ошибка/предупреждение) различимы не только цветом
- [ ] Формы — у каждого input есть связанный `<label>` (не только placeholder)
- [ ] Скринридер (VoiceOver/NVDA) — пройти основной флоу (регистрация → добавление чека) и услышать, понятно ли, что происходит
