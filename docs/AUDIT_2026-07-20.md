## 2. Неточность в AUDIT_2026-07-19.md, которую нужно поправить

### 2.3 "Приватный ключ магазина в sessionStorage" — **на самом деле уже не так**
Открыл `business-panel.js:223` и `:318` — оба места, где формируется объект `newShop`/`shop`
перед `sessionStorage.setItem('current_shop', ...)`, используют
`.select('id, shop_name, tax_id, address, public_key, owner_id')` — **`private_key` в выборке нет**.
Приватный ключ загружается отдельно (`.select('private_key')`, строка 422) только в момент
подписи чека и не кладётся в sessionStorage. Пункт можно закрывать как "не воспроизводится".

## 3. Подтверждённые открытые проблемы (перепроверено построчно сегодня)

### Критично

**Unhandled promise rejections — все 4 ещё живы:**
- `dashboard-settings.js:322` — `ensureSettingsLoaded()` зовёт `initSettings()` (async) без await/catch
- `dashboard-notifications.js:153,157` — `ensureNotifLoaded` как click-handler и на строке 157 вызывается напрямую без `.catch()`
- `business-panel.js:846` — `initBusinessPanel();` на верхнем уровне модуля, без обработки ошибок

Если Supabase временно недоступен при загрузке — пользователь просто видит "зависшую" вкладку без сообщения об ошибке, потому что `catch` нигде нет.

**CDN-глобалы без защиты — подтверждено в verify.js:**
```
verify.js:140  window.jsQR(...)       — нет проверки typeof
verify.js:148,622  pdfjsLib.getDocument — нет проверки typeof
verify.js:428  window.jsQR(...)
```
Если `cdnjs`/jsdelivr недоступен (блокировщик, региональная блокировка, сбой CDN) — вся страница верификации чека падает с `ReferenceError` без единого сообщения пользователю. Это единственная страница, где чужой человек (не залогиненный) проверяет подлинность чека — она обязана деградировать красиво.

### Средне

- **Несогласованность бага с гарантией `0` месяцев:** `dashboard-items.js:68` уже пофикшен (`months == null`), а **`dashboard-notifications.js:13`** — нет, там всё ещё `if (!purchaseDate || !months) return -999;`. Результат: товар с `warranty_months = 0` корректно отображается на вкладке "Мои товары", но в уведомлениях его гарантия трактуется как отсутствующая. Разные части приложения по-разному понимают один и тот же товар.
- **`business-panel.js:49`** — `r.id.slice(0, 8)` без `String()`, в отличие от уже исправленного аналога в `receipts.js:232`. Если `id` когда-либо придёт не строкой — упадёт именно бизнес-панель.
- **Открытый редирект** через `?from=` в `verify.js:692` подтверждён — `backBtn.href = fromParam` без проверки, что это внутренний путь. Можно оформить фишинговую ссылку вида `valuon.app/verify.html?from=https://fake-valuon.com`, и кнопка "Назад" уведёт на чужой домен.
- **Слабый `sanitizeHTML`** в `script.js:137-142` — режет только `<script>`, `on*=`, `javascript:`. Сейчас безопасно, потому что через него проходят только захардкоженные строки переводов, но это мина замедленного действия: как только через `data-i18n`-конвейер пойдут пользовательские данные, будет XSS.
- **Слабая политика пароля:** `register.js:48` — только `password.length < 6`. Функция `validatePassword` существовала в `security.js`, но её удалили как "мёртвый код" в прошлом аудите (3.1) вместо того чтобы подключить в register.js — по сути, минимальную защиту снесли, а не усилили.
- **6 независимых мест создания Supabase-клиента** (`auth.js`, `dashboard-auth.js`, `register.js`, `script.js`, `verify.js`, `reset-password.html`) — ротация anon-ключа требует правки в 6 местах, велик риск рассинхрона.

### Низко
- `.sidebar-overlay { display: none !important; }` (`dashboard.css:741`) действительно перебивает `.sidebar-overlay.active` (`dashboard.css:1280`) на `max-width: 900px` — оверлей за сайдбаром физически не появится на мобильных, хотя JS честно навешивает `.active`.
- Дублирование Supabase credentials в inline-скрипте `reset-password.html` вместо переиспользования модуля.

## 5. Продуктовые рекомендации — что добавлять дальше

Технический бэклог (легализация в ЕС, RLS, реальная Ed25519-подпись) уже подробно расписан в
`docs/Valuon_Estonia_Backlog.md` — не дублирую. Здесь — функциональность поверх уже работающего MVP,
в порядке ценности для пользователя/бизнеса:

### Для покупателей
- **OCR-автозаполнение чека по фото** — сейчас это прямо на лендинге помечено "в разработке" (`feat_1_desc`). Это самое частое ручное действие в приложении (добавление товара) — снятие трения здесь даёт наибольший прирост удержания.
- **Экспорт для возврата/спора** — один PDF со всеми чеками по конкретному магазину/периоду, чтобы приложить к претензии или налоговому вычету.
- **Шаринг чека без раскрытия остальных покупок** — сейчас, чтобы показать один чек (например, риелтору, страховой), нет ссылки "поделиться только этим чеком".

### Для бизнеса
- **Ротация/отзыв ключей магазина** (уже отмечено M10 в CODE_AUDIT) — без этого компрометация ключа необратима навсегда.
- **Webhook/API для интеграции с кассой (POS)** — сейчас чек создаётся вручную в бизнес-панели; для реального магазина с потоком продаж это неприемлемо, нужен хотя бы простой REST-эндпоинт "создать чек".
- **Массовая выгрузка чеков** (CSV/XLSX за период) — бухгалтерии магазина нужен экспорт для отчётности, сейчас только просмотр по одному.

### Инфраструктурное, но заметное пользователю
- **Статус-страница/деградация при недоступности CDN** — см. пункт 2 из технического списка: `verify.html` не должна быть "белым экраном" для стороннего человека, который просто сканирует QR в магазине.
- **Push/email-уведомление о начале действия гарантии**, а не только "скоро истекает" — сейчас есть только напоминания об окончании (30/7/1 день), но нет подтверждения "чек принят, гарантия активна с такого-то числа".

Если хочешь, могу взять любой из пунктов раздела 4 (технический) и сразу починить в коде — 1, 5 и 6 займут пару минут.
