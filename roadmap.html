Дорожная карта Valuon: единая платформа официальных чеков

План развития от рабочего MVP до единой платформы доступа к официальным чекам в разных странах.

Две роли Valuon, а не одна универсальная

Роль А

Первичный эмитент

Эстония, Ирландия, Мальта, Финляндия, Лихтенштейн, США — там, где нет обязательной фискализации, чек Valuon может быть настоящим официальным цифровым чеком, а не имитацией.

Роль Б

Реестр поверх официального

Германия, Франция, Италия, Испания и другие — чек уже подписан государственно сертифицированной кассой. Valuon подключается к этим данным и даёт чеку вторую жизнь у покупателя.

Роль В — отложено

Собственная сертификация

Стать сертифицированным фискальным провайдером самим (как fiskaly/efsta) — дорогой многолетний путь. Решение принимается не раньше 2028 года, под подтверждённый масштаб.

Гарантии, история владения и перепродажа техники работают поверх обеих ролей как надстройка — они делают возврат в приложение осмысленным, но не являются целью сами по себе.

Роль А + Роль Б — основа

MVP: текущий этап — что уже сделано

Рабочий продукт, который можно открыть, зарегистрироваться и начать использовать. Дальнейшие этапы строятся поверх этой базы.

  • Регистрация с email/паролем, подтверждение email, восстановление пароля
  • Проверка от ботов при входе и регистрации
  • Смена email, редактирование имени/фамилии в настройках
  • Добавление вещи вручную: название, тип, бренд, цена, серийный номер, дата покупки, срок гарантии
  • Автоматический расчёт даты окончания гарантии
  • Календарь гарантий с переключением месяцев
  • Раздел уведомлений: гарантия ≤30 дней или истекла
  • Загрузка фото/PDF чека, drag&drop, привязка к вещи
  • Бизнес: создание магазина, генерация Ed25519 ключей
  • Бизнес: выписка чека с несколькими позициями и цифровой подписью
  • Бизнес: PDF с QR-кодом, график продаж
  • Верификация: сканирование QR камерой, загрузка PDF, проверка Ed25519
  • Тёмная/светлая тема, мультиязычность RU/EN

Этап 1: доработка технической базы

Прежде чем звать больше магазинов и пользователей — нужно укрепить то, что уже построено.

  • Выгрузить и закоммитить RLS-политики в SQL-миграции
  • Подключить проверку сложности пароля на регистрации и сбросе
  • Добавить CSP-заголовки (vercel.json)
  • Настроить проверку MIME-типа файлов в Storage bucket
  • Добавить версию формата в подписываемые данные чека
  • «Заморозить» данные магазина на чеке при проверке подписи
  • Починить округление сумм в PDF
  • Ограничения CHECK на значения позиций чека в БД
Без этого этапа масштабировать нельзя — рост числа магазинов и чеков увеличит ущерб от уже найденных проблем.
Месяцы 1–3
Роль А + активация

Этап 2: активация покупателя — retention D30

Ручной ввод чека — главный барьер возврата. OCR, импорт из почты и push-уведомления должны сделать приложение привычкой, а не одноразовым экспериментом.

  • Автозаполнение полей вещи по фото чека (OCR)
  • Импорт чеков из почты (Amazon, Ozon, AliExpress и др.)
  • Push-уведомления об истечении гарантии
  • Фактическая отправка еженедельной рассылки (cron / Edge Function)
  • Добавление даты окончания гарантии в Google/Apple Calendar
  • Журнал сервисного обслуживания вещи (ремонты, даты, стоимость)
  • Мастер гарантийного обращения: письмо продавцу со ссылкой на чек
  • Передача вещи новому владельцу при перепродаже (QR с историей)
  • PWA-версия с камерой в один тап
Месяцы 3–6
Роль Б + бизнес-инструменты

Этап 3: инструменты для бизнеса

Магазины не будут выписывать чеки вручную. Нужны интеграции с кассами, API и бесшовный онбординг — только тогда выписка чека станет естественной частью продажи.

  • Плагины к кассам и e-commerce (Shopify, WooCommerce, Lightspeed)
  • Публичный REST API для программной выписки чеков
  • Вебхуки на события (чек выписан, чек верифицирован)
  • Мультипользовательские аккаунты магазина (роли сотрудников)
  • Кастомный брендинг чека и страницы верификации
  • Массовый импорт исторических чеков через CSV
  • Workflow возврата/обмена/аннулирования чека
  • Ротация и отзыв криптоключей магазина
  • Экспорт чеков для бухгалтерии (CSV/XML под 1С, DATEV)
  • Расширенная аналитика: LTV, средний чек, доля гарантийных обращений
Месяцы 6–12
B2B2C-инфраструктура

Этап 4: монетизация и сетевой эффект

Сейчас в продукте нет платёжного функционала вообще. Первые тарифы — только после того, как пилотные магазины подтвердят готовность платить.

  • Тарифные планы и биллинг (сейчас платёжного функционала нет)
  • Публичный виджет «Verified by Valuon» для сайта магазина
  • Платный API верификации для страховых и площадок перепродажи
  • Продажа расширенной гарантии в момент выписки чека (комиссия)
  • Онбординг-чеклист для нового магазина в дашборде
Платный тариф запускается только после письменного подтверждения магазинов, а не одновременно с пилотом.

Этап 5: географическое и юридическое расширение

Из одной страны и одной роли — в несколько юрисдикций. Каждый новый рынок запускается только после измеримого успеха на предыдущем.

  • Пилот Роли А в Эстонии (5–10 продавцов) — Valuon как их единственная система чеков
  • Параллельный точечный пилот Роли А в США
  • Первая интеграция Роли Б: коннектор к TSE/кассовому API (Германия)
  • Юридическая консультация: можно ли называть импортированный чек «официальным»
  • Выбор основной юрисдикции для юрлица (Эстония)
  • Расширение Роли А на Ирландию/Финляндию при успехе Эстонии
  • Расширение Роли Б на Францию (сертификация касс с сентября 2026)
  • Решение по Роли В — только под подтверждённый спрос, не раньше 2028
Ключевая метрика: Retention D30 ≥ 25%. Если retention низкий — рассмотреть встраиваемый виджет вместо отдельного приложения.
